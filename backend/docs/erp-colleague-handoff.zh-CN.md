# AI ERP Assistant × 成熟 ERP 联调说明（发给 ERP 侧同事）

**文档用途**：统一说明双方职责、推荐推进顺序，以及 **ERP 团队需要准备哪些材料**；便于评审排期与一次性收齐资料。  
**技术细节补充**：仓库内另有 `backend/docs/erp-integration-spec.zh-CN.md`（路径、字段、错误码更细）。

---

## 1. 双方系统在链路上的分工

| 角色 | 负责内容 |
|------|----------|
| **ERP / 业务中台** | 主数据查询（供应商、物料等）；单据 **校验草稿**、**创建草稿**；稳定错误码；**幂等**建单；沙箱与测试数据。 |
| **AI ERP Assistant（调用方）** | 上传与解析文档、结构化抽取、缺字段时让人补全、映射到 ERP 编码；调用贵方接口完成校验与建草稿；记录审计日志（**不落库替代 ERP 当事实源**）。 |

**说明**：专业问答若也由贵司网关或 ERP 只读接口提供，可与「录单链」并行，本阶段 **录单 MVP 不依赖** 通用知识库或大模型即可联调。

---

## 2. 推荐推进顺序（减少返工）

1. **先**：沙箱环境 + 鉴权文档 + **主数据搜索**（至少供应商、物料）。  
2. **再**：**校验草稿** + **创建草稿**（含 `idempotency_key` 语义）。  
3. **并行**：PO / GR / INV（或贵司等价单据类型）的 **字段映射表** 与 **必填规则**。  
4. **可选**：仓库、税码等搜索（映射与校验需要时再开）。  
5. **最后**：生产网络策略、限流、监控与变更窗口（不影响沙箱先跑通）。

---

## 3. 请 ERP 团队统一准备以下交付物（勾选清单）

请尽量按 **「一份 OpenAPI + 一份联调包」** 交付，避免信息散落在邮件里。

### 3.1 环境与账号

- [ ] **沙箱 Base URL**（HTTPS）；若「主数据查询」与「写单服务」分属两套系统，请分别提供两个根地址（或说明共用一个网关）。  
- [ ] **测试账号**（至少一套）：含调用上述接口的权限。  
- [ ] **测试组织**：如 `org_id` / 公司代码及对应说明（调用方会在请求中带组织维度）。  
- [ ] **网络**：是否需要 VPN / 固定出口 IP / mTLS；若有 API 网关，请给 **路径前缀** 说明。

### 3.2 鉴权与安全

- [ ] **鉴权方式说明**：如 Bearer Token、API Key Header、OAuth2 客户端模式、自定义签名等。  
- [ ] **Token 获取**：授权 URL、请求参数、响应中 token 字段名、**有效期与刷新方式**。  
- [ ] **若需签名**：算法、参与签名的字段、时间戳容忍度、**1 个可复制的签名示例**（含请求与期望结果）。  
- [ ] **权限模型**：按组织/角色隔离方式；无权时的 HTTP 状态与 **业务错误码**。

### 3.3 接口文档（强烈建议 OpenAPI 3）

至少需要覆盖下面 **「第 4 节接口清单」** 中的全部 **必选** 接口；路径可与下表不一致，但请在 OpenAPI 中写清 **真实路径与方法**。

### 3.4 业务契约

- [ ] **单据类型**：是否与本项目约定的 `PO` / `GR` / `INV` 一一对应；若不对应，请给 **类型映射表**（例如：贵司 `PurchaseRequisition` → 我方逻辑中的 `PO`）。  
- [ ] **字段映射**：从「校验/建草稿请求体」角度，提供 **JSON Schema 或示例报文**（脱敏）；并标注 **必填 / 条件必填 / 枚举**。  
- [ ] **幂等**：创建草稿时 **幂等键** 放在 Header 还是 Body、字段名、重复提交时 **HTTP 状态 + 响应体**（是否返回同一 `draft_no`、是否带 `idempotent_replay` 类标记）。  
- [ ] **主数据搜索**：是否支持模糊、分页、条数上限；列表字段命名（若与下文「响应字段名」不同，请在映射表中写出）。

### 3.5 错误码与排障

- [ ] **业务错误码清单**（字符串常量）：含义、可重试建议、对应 HTTP 状态习惯。  
- [ ] **request_id / trace_id**：是否返回、字段名（便于双方对账）。  
- [ ] **脱敏要求**：日志或审计中 **禁止回传** 的敏感字段列表（如个人数据、完整税号等）。

### 3.6 联调样例包（便于一次验收）

- [ ] 每个核心接口至少 **1 组成功** + **1 组失败** 示例（请求 + 响应，可脱敏）。  
- [ ] **主数据**：保证沙箱内存在可搜到的测试供应商、测试物料（及若启用：仓库、税码）。  
- [ ] **校验失败样例**：能稳定返回缺字段或 `violations`（便于我方补全 UI 联调）。

## 4. 接口清单（MVP 必选 / 可选）

以下为调用方当前适配层约定的 **能力名称**；贵方实现可用自有 URL，通过 OpenAPI 对齐即可。

| 能力 | 必选 | 调用方默认路径（可改配置对齐贵方） | 请求要点 | 成功响应要点 |
|------|------|-----------------------------------|----------|--------------|
| 供应商搜索 | **必选** | `GET` 或 `POST`：`/erp/vendors/search` | `org_id`、`keyword` | 列表项含 `vendor_code`、`vendor_name`（字段名可通过配置映射） |
| 物料搜索 | **必选** | `GET` 或 `POST`：`/erp/materials/search` | 同上 | `material_code`、`material_name` |
| 仓库搜索 | 可选 | `/erp/warehouses/search` | 同上 | `warehouse_code`、`warehouse_name` |
| 税码搜索 | 可选 | `/erp/tax-codes/search` | 同上 | `tax_code`、`tax_name` |
| 校验草稿 | **必选** | `POST`：`/erp/drafts/validate` | `type`（`PO`/`GR`/`INV`）、`payload` | `valid`；失败时 `missing_fields` 与/或 `violations` |
| 创建草稿 | **必选** | `POST`：`/erp/drafts` | `type`、`payload`、**`idempotency_key`** | `draft_no`、建议 `draft_url` |

**双宿主说明**：若主数据与写单分属不同域名，请分别提供两个 Base URL；调用方可分别配置（与仓库 `.env.example` 中 `ERP_DATA_BASE_URL`、`ERP_WRITE_BASE_URL` 对齐）。

## 5. 单据 `payload` 最小字段参考（MVP）

贵司字段名不必一致，但请给 **对照表** 或 **转换规则**。以下为调用方当前「最小模型」参考，便于对齐评审。

**公共头（示例）**：组织/公司、`vendor_code`、`doc_date`、`currency`、`tax_included`、以及用于审计的 `source_file_id` / `source_file_hash`（若贵方不需要可说明忽略策略）。

**按类型（行级示例口径）**：

- **PO**：`material_code`、`qty`、`uom`、`price`、`tax_code`、`delivery_date`  
- **GR**：`po_no`、`po_line_no`、`material_code`、`qty_received`、`warehouse_code`  
- **INV**：`invoice_no`、`invoice_date`、`po_no` 或 `gr_no`、`material_or_expense_code`、`qty`、`unit_price`、`tax_code`

若贵司 **行项目为多行数组结构**，请直接给出 **目标 JSON 样例**（脱敏），调用方在适配层做映射。

## 6. 建议的业务错误码（请贵司对齐或给出等价映射）

| 建议码 | 含义 |
|--------|------|
| `MASTER_DATA_NOT_FOUND` | 主数据不存在（如供应商/物料无效） |
| `MISSING_REQUIRED_FIELDS` | 缺必填业务字段 |
| `PERMISSION_DENIED` | 无权限 |
| `AMOUNT_NOT_BALANCED` | 金额/税额不平 |
| `DUPLICATE_REQUEST` | 重复或冲突提交 |
| `UPSTREAM_TIMEOUT` | 上游超时 |
| `SYSTEM_ERROR` | 系统错误 |

若贵司已有成熟错误码体系，请提供 **「贵司码 → 上表或我方处理策略」** 对照即可。

---

## 7. 联调验收标准（双方共同确认）

1. 供应商、物料搜索在沙箱可返回 **非空候选**。  
2. `validate` 在缺字段时能返回 **明确缺失项**（或结构化 `violations`）。  
3. `create` 首次调用返回 **草稿号**（及可选跳转 URL）。  
4. **相同 `idempotency_key` 重放** 不重复建业务草稿（行为与文档一致）。  
5. 主数据类错误返回 **稳定业务错误码**，便于调用方映射与审计。

## 8. 调用方（AI ERP Assistant）会向贵方提供 / 约定的事项

便于 ERP 同学评估接口设计是否够用：

| 项目 | 说明 |
|------|------|
| 调用身份 | 按贵方要求携带 Token / 签名；组织维度一般传 `org_id`（与贵方字段名对齐即可）。 |
| 幂等 | 同一业务 ingestion 建草稿会使用 **稳定幂等键**；请贵方保证语义与文档一致。 |
| 只读事实 | 库存、订单量、金额等 **以贵方实时查询/校验结果为准**；调用方不把 ERP 事实写入外部知识库当真相源。 |
| 配置对齐 | 若响应 JSON 字段名与默认不同，调用方可通过环境变量配置项对齐（详见仓库 `.env.example` 中 `ERP_*` 段）。 |

---

## 9. 资料提交方式建议

请 ERP 侧尽量打包为：

1. **OpenAPI 3 YAML/JSON**（含沙箱 `servers`）；  
2. **联调说明 PDF/Markdown 一页**：鉴权步骤 + 幂等说明 + 错误码表；  
3. **样例目录**：`examples/` 下按接口分子目录，每接口 `success.json` / `failure.json`。
